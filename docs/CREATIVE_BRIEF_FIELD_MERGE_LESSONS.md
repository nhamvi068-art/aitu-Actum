# Creative Brief 字段合并经验

## 背景

短视频工具里的「视频用途」和「短视频场景」语义接近，用户需要在两个下拉框里重复做相似决策。项目尚未发布时，可以优先收敛数据模型，避免为了兼容未公开字段继续背负复杂度。

## 经验

- 未发布字段可以直接删除，但要一次性同步类型、UI、prompt formatter、测试和 OpenSpec，避免出现“界面删了、prompt 还在用”的半迁移状态。
- 当两个字段都能回答“这条视频要往什么方向做”时，优先合成一个业务决策字段，例如「视频用途/场景」。
- 合并预设时不要简单拼接两个数组，要顺手去重和重分组，让用户看到的是一个清晰选择，而不是两个旧下拉框塞进一个菜单。
- 删除字段后，`normalize` 应使用字段白名单，不要遍历输入对象的所有 key，否则旧缓存或历史记录里的废弃字段仍可能被保存回去。
- Prompt 文案也要跟着改概念：从“用途决定目标、场景决定形态”改成“用途/场景同时决定钩子、镜头形态、信息密度和 CTA”。
- 测试断言要检查新概念是否进入摘要和生成 prompt，同时用搜索确认旧字段名、旧标签和旧配置没有残留。

## 推荐流程

1. 先查字段的所有引用，确认它是否只在当前功能边界内使用。
2. 删除类型字段和 UI 控件，只保留合并后的字段。
3. 将旧预设迁入新预设分组，减少重复选项。
4. 用白名单 normalize 丢弃旧字段。
5. 更新 OpenSpec、单测、prompt 断言。
6. 跑定向测试、库级 typecheck、`git diff --check`。
