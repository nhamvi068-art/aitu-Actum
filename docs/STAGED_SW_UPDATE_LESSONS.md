# Service Worker 分阶段升级经验总结

更新日期：2026-04-23

## 现象

之前版本升级时，用户进入页面后经常会落在“新资源正在接管”的窗口里，表现为：

- 老用户刚打开页面，就开始碰新版本资源
- `Service Worker` 已经切到新版本，但新版本资源还没完全准备好
- 页面会卡在启动中间态，或者撞上旧入口引用失效 hash 的问题

这类体验最差的点在于：

- 老用户明明本地已经有一份可运行的旧版本缓存
- 却没有先稳定进入旧版本
- 而是被升级链路直接打断

## 根因

### 1. 更新版本在 install 阶段就立刻接管

之前的升级逻辑核心是：

- `install` 阶段直接 `skipWaiting()`
- `activate` 阶段直接 `clients.claim()`

这对“首次安装的新用户”是有利的，因为能更快拿到缓存能力。  
但对“已有旧版本缓存的老用户”是反效果：

- 当前会话还在跑旧版本
- 新 worker 却立刻接管页面
- 结果当前会话被硬切到尚未准备完的新版本链路

一句话：  
**首次安装和版本升级不能共用同一套 SW 接管策略。**

### 2. “已发现新版本”和“新版本已可切换”被混为一谈

之前主线程里只要看到：

- `registration.updatefound`
- 或 worker `installed`

就会倾向于提示“可以升级”。

但这只说明：

- 新 worker 已经下载好了主体脚本

不代表：

- 新版本依赖的整包静态资源已经缓存齐
- 用户点击更新后就一定能顺滑切过去

一句话：  
**worker 已安装，不等于新版本已 ready。**

## 更稳的升级模型

### 1. 新用户：最新版 + CDN 优先

新用户没有旧缓存包袱，目标是：

- 尽快进入当前最新版
- 首屏静态资源仍然走 `CDN 优先 + 源站兜底`
- SW 尽快启用缓存能力

所以首次安装仍然允许：

- `skipWaiting`
- `clients.claim`

### 2. 老用户：锁旧版本运行

老用户进入页面时，最优先目标不是升级，而是：

- 先稳定进入手头已有的旧版本
- 当前会话不要被中途切断

所以在“版本升级”场景下：

- 新 worker 安装后保持 `waiting`
- 不立刻 `skipWaiting`
- 不立刻 `clients.claim`

这样当前 tab 就能继续跑旧版本，避免进入半切换状态。

### 3. 新版本后台整包准备

等待中的新 worker 可以在后台继续做：

- 读取预缓存清单
- 拉取新版本核心静态资源
- 写入新版本对应的静态 cache

只有当这些关键资源真正准备完成后，才能认为：

- 新版本已经可以安全接管

### 4. 用户确认后再切版本

正确的交互顺序应该是：

1. 用户先用旧版本正常工作
2. 后台把新版本整包准备好
3. UI 提示“新版本已就绪”
4. 用户点“立即更新”
5. waiting worker 再 `skipWaiting`
6. 激活后 reload，切入新版本

这才是“升级提示”和“升级执行”的正确分层。

## 这次代码层面固化的规则

### 0. 升级协议必须有“待提交”和“已提交”两个阶段

只靠下面这些信号不够：

- `updatefound`
- `installed`
- `sw:newVersionReady`

因为这些信号都只能说明：

- 新 worker 出现了
- 或者新 worker 的资源大体准备好了

但还不能说明：

- 导航入口已经允许切到新版本
- 新 tab 可以安全直接吃到新版本 HTML

这次更稳的做法是引入两类版本状态：

- `committedVersion`：当前正式对页面生效的版本
- `pendingVersion`：后台已经预热完成、但尚未被用户确认切换的版本

再配一个显式状态机：

- `idle`
- `prewarming`
- `ready`
- `committing`

一句话：  
**升级不是“发现新版本就切”，而是“先后台预热，再显式提交版本”。**

### 1. install 阶段区分首次安装和版本升级

应该先判断：

- 当前注册里是否已经存在 active worker

如果没有：

- 说明是首次安装
- 可以立即接管

如果有：

- 说明是版本升级
- 不能立刻接管当前页面

### 2. activate 阶段是否 `claim` 必须可控

`activate` 不是天然就该 `clients.claim()`。

更稳的做法是：

- 首次安装：允许 claim
- 版本升级：默认不 claim
- 只有用户确认升级后，才允许新 worker 接管页面

### 3. 更新提示只由“整包 ready”触发

不要在这些时机直接弹更新提示：

- `updatefound`
- `installed`
- `activated`

更合理的是：

- 只有后台关键静态资源预缓存完成
- 再发送 `new version ready`

也就是说，提示代表的是：

- **可以切**

而不是：

- **发现了新版本**

### 4. 更新提示不能只靠广播，必须可补读

如果提示链路只依赖一次性广播，就会有两个经典丢失场景：

- 页面还没完成 duplex channel 初始化
- 页面已经打开，但当时没有订阅对应事件

结果就是：

- 后台其实已经把新版本缓存好了
- 但用户永远看不到“立即更新”

这次更稳的默认策略应该是：

- SW 把 `pendingVersion` 持久化到本地状态
- 页面启动后主动发送 `GET_VERSION_STATE`
- 即便错过广播，只要 `pendingVersion` 还在，也能重新弹出升级提示

一句话：  
**升级提示属于状态，不属于瞬时事件。**

### 5. 新 tab 在未确认升级前必须继续跑 committed 版本

这次线上暴露得最明显的问题不是“旧 tab 升级失败”，而是：

- 老 tab 还没提示升级
- 新 tab 已经直接拿到新版本 HTML
- 新 hash 资源还在慢慢拉
- 用户感知就是打开特别慢，像资源没准备好

这说明仅仅“waiting worker 不 claim”还不够，  
还必须补一条更强的规则：

- **navigation 请求只允许命中 `committedVersion`**
- **`pendingVersion` 只负责后台预热，不负责前台入口切换**

否则就会出现：

- 老 tab 继续旧版本
- 新 tab 却偷偷吃到新版本

版本体验分裂。

一句话：  
**版本切换的边界不在 worker 安装完成，而在 committedVersion 被正式提升。**

## 2026-04-24：更新按钮“点了没升级”这类问题的经验

### 现象

这轮线上暴露出的表象是：

- 更新条已经提示“新版本已就绪”
- 用户点击“立即更新”后，提示条直接消失
- 页面却还是老版本，用户会误以为按钮没生效

另一类更隐蔽的表现是：

- 页面确实开始进入升级链路
- 但 reload 触发得太早
- 刷新时新的 `Service Worker` 还没真正 `activate + claim`
- 结果刷新后依旧停在旧版本

一句话：
**“提示消失”不等于“版本已切换”。**

### 根因

#### 1. `COMMIT_UPGRADE` 不能兜底发给旧 `controller`

之前主线程在找不到 `waiting worker` 时，容易退化成：

- 给当前 `controller` 发 `COMMIT_UPGRADE`

这会出现一个非常危险的假成功：

- 老 worker 收到了消息
- UI 状态被清掉了
- 但真正等待接管的新 worker 并没有被提交

结果就是：

- 用户看见更新条消失
- 实际版本没有切过去

一句话：
**升级提交消息只能发给真正的 `waiting / installed worker`，不能发给正在服务页面的旧 `controller`。**

#### 2. `sw:updated` 太早，不适合作为 reload 依据

这轮还验证了一个很关键的时序问题：

- `waiting worker` 收到 `COMMIT_UPGRADE`
- 调用 `skipWaiting()`
- 如果这时立刻广播“已更新”
- 主线程按这个信号 1 秒后 reload

那 reload 很可能发生在：

- `activate` 还没完成
- `clients.claim()` 还没完成
- 版本状态还没完全落盘

这时用户就会遇到：

- 点了更新
- 页面也刷新了
- 但刷新后还是旧版本

一句话：
**`skipWaiting()` 只是“开始切”，不是“已经切完”。**

#### 3. 更新条不能在点击时立即隐藏

更新条按钮如果在点击瞬间就：

- `setUpdateAvailable(null)`

那一旦后续提交失败，用户会进入最糟糕状态：

- 看不到更新条了
- 版本也没升级
- 同版本提示又可能因为去重逻辑不再重复弹出

更合理的交互是：

- 点击后只发升级请求
- 提示条保持可见，或最多只关更新详情弹窗
- 等真正接管完成后，通过 reload 自然消失

一句话：
**升级提示的关闭时机应该跟“接管成功”绑定，而不是跟“用户点击了按钮”绑定。**

#### 4. idle prefetch manifest 失败不能永久卡死

这轮另一个会间接影响升级体验的问题是：

- `idle-prefetch-manifest.json` 偶发返回首页 HTML / 非法 JSON / 临时失败
- SW 如果把这次失败永久缓存成 `null`
- 后续 idle prefetch 就再也不会继续

这样会导致：

- 页面表面上已经进入 staged update
- 但按需静态资源没有补齐
- 最终 `ready` 时机不准确，或者某些离线场景仍然缺 chunk

更稳的做法是：

- 对 manifest 失败做明确日志
- 识别 HTML fallback / 非法 JSON
- 不永久记住这次失败
- 按固定冷却时间重新拉取并恢复队列

一句话：
**清单加载失败要“可恢复”，不能“一次失败，永久失忆”。**

### 这次应该固化成默认规则的做法

- 升级提交只发给 `waiting / installed worker`，绝不发给旧 `controller`
- 主线程 reload 只等真正接管完成后的信号：
  `sw:activated` 或 `controllerchange`
- `sw:updated` 只可用来同步状态，不能直接驱动 reload
- 更新条点击后不要提前隐藏，失败时必须保持可见或可恢复
- idle prefetch manifest 失败后必须带冷却重试，不能把 `null` 永久缓存

### 后续排障顺序建议

1. 先看 `COMMIT_UPGRADE` 最终发给了谁，是 `waiting worker` 还是旧 `controller`
2. 再看页面是被什么事件触发 reload，是 `sw:updated` 还是 `controllerchange / sw:activated`
3. 再看更新条是否在点击瞬间就被本地状态清掉
4. 最后看 `idle-prefetch-manifest` 是否缺失、回了 HTML、或被失败缓存住

### 一句话总结

这轮最重要的不是“把按钮点通”，而是明确了升级链路的 4 条边界：

- 提交对象必须对
- reload 时机必须晚
- UI 关闭时机必须更晚
- manifest 失败必须能恢复

只要这 4 条不破，`staged update` 才不会从“平滑升级”退化成“假升级”。 

### 6. 静态缓存元数据必须带“所属版本”，不能默认写当前 SW 版本

这次实现里还有一个很容易忽略的坑：

- SW 当前脚本版本是新版本
- 但它可能正在给旧版本 `committedVersion` 回填缓存

如果此时缓存条目头部总是写：

- `x-sw-app-version = APP_VERSION`

那旧版本缓存会被误标记成“新版本资源”，后续清理和命中判断都会乱掉。

所以缓存元数据必须显式区分：

- 当前正在运行的 SW 版本
- 当前缓存条目所属的应用版本

一句话：  
**缓存条目的版本归属要跟“服务对象版本”一致，而不是跟“执行者 SW 版本”一致。**

### 7. 首屏加载进度不能把 50% 当硬上限

启动条长期卡在 `50%` 的问题，本质不是单个请求慢，而是：

- 前半段是模拟进度
- 后半段没有足够强的真实阶段信号去接管 UI

如果模拟逻辑在 `50%` 就停住，用户就会自然理解成：

- 页面卡死了

更稳的做法是：

- 模拟进度只作为早期兜底，不做过低的硬上限
- 主线程在 SW 连接、运行时恢复、React 挂载等关键阶段主动推进进度
- App ready 时再收口到 100%

另外，`sw.js` 如果同时：

- `preload`
- 又 `register('/sw.js')`

新用户首屏就很容易看到重复请求。

一句话：  
**启动页要让真实阶段接管进度，别让模拟进度长期占着主舞台。**

## 关于版本号和缓存命名的经验

### 1. SW 缓存命名必须跟版本自动变化

缓存命名不跟版本变，会带来两类问题：

- 新旧版本静态资源混用
- 老入口拿到新资源、新入口拿到旧缓存

现在项目里比较正确的做法是：

- `APP_VERSION` 来自 `version.json`
- 构建时注入到 `sw.js`
- `CACHE_NAME` / `STATIC_CACHE_NAME` 自动带版本号

一句话：  
**缓存版本隔离应该自动依赖发版号，而不是手工维护。**

### 2. 发版时要确保版本号真的 bump 了

如果版本号没变：

- `sw.js` 内部版本不会变
- cache 名也不会变
- 浏览器就可能把这次发布当成同一版本

所以不能只“发代码”，还要保证：

- 发版号同步变化

## 适合长期固化的默认策略

- 新用户：最新版、CDN-first、SW 立即启用
- 老用户：锁当前 active 版本运行
- 更新 worker：后台整包准备，不打断当前会话
- 更新提示：只在整包 ready 后显示
- 用户确认后：再 `skipWaiting + reload`
- cache 命名：始终随版本自动变化
- 导航入口：始终只认 `committedVersion`
- 升级提示：必须支持页面启动后主动补读
- 启动条：模拟兜底，真实阶段接管

## 一句话总结

`Service Worker` 升级的核心不是“尽快接管”，而是：

- 新用户尽快接入最新版
- 老用户当前会话绝不被升级过程打断

真正稳的模型不是“发现新版本就切”，而是：

**旧版本先服务，新的版本后台准备，准备好了再让用户决定何时切换。** 🔄

## 2026-04-23：启动假卡死与工具资源拆组经验

### 现象

- 启动条会长时间卡在 `68%`
- 网络面板里看到 `tool-windows-*.js` 很慢，容易误判成“首屏核心资源没下完”
- 改完 chunk 分组后，部署检查脚本和校验脚本还在找旧的 `tool-windows-*`
- 类型定义已经换了，但局部逻辑或脚本还保留旧分组名，容易报：
  `不能将类型“..."”分配给类型“IdlePrefetchGroup”`

### 根因

#### 1. `68%` 卡住，很多时候不是功能真阻塞，而是启动条写法在制造错觉

这次真实问题不是：

- React 壳子一定在等某个大包

而是：

- HTML 启动页的模拟进度硬停在 `68%`
- 后续必须等主线程更高阶段信号才能继续推进
- 偏偏同一时段 `tool-windows` 这类延后资源又开始抢网络

于是用户的体感就变成：

- 进度卡在 `68%`
- 某个大 JS 正在慢慢下
- 所以一定是这个大 JS 把首页卡住了

一句话：  
**启动页的固定百分比上限，会把“网络慢”伪装成“首屏依赖没准备好”。**

#### 2. `tool-windows` 被过度打包，导致“点一个入口，吞一整包”

之前 `tool-windows` 里混了三类东西：

- 运行时增强：任务恢复、字体预热、资产集成
- 抽屉/WinBox：项目抽屉、工具箱、知识库工具窗
- 对话框/低频 UI：媒体库、同步、备份恢复、版本提示、性能面板、命令面板、搜索

结果是：

- 首屏一旦误触发默认 idle prefetch，就会把整个大组提早拉起
- 用户只打开一个抽屉，也会顺带背上其它完全不相关的模块

一句话：  
**按“都是工具功能”打一个大组，最终一定会反噬启动和首开交互。**

#### 3. chunk 分组名不是只改一处就结束，而是“类型、构建、测试、校验”全链路资产

这次拆组后，真正需要一起改的地方不只业务代码，还包括：

- `IdlePrefetchGroup` 类型定义
- Vite `manualChunks` / `idle-prefetch-manifest` 生成逻辑
- 页面侧触发预取与监听完成状态的逻辑
- 部署包检查脚本
- 启动入口校验脚本
- CDN fallback 测试里对 chunk 文件名的示例

如果只改前 2~3 处，就很容易出现：

- 代码能跑
- 但打包检查仍报缺少 `tool-windows-*`
- 或测试仍引用旧 chunk 名

一句话：  
**chunk 分组名属于“发布契约”，不是普通内部变量。**

### 更稳的做法

#### 1. 启动进度不要设过低硬上限

更稳的规则是：

- 早期允许模拟进度兜底
- 但不要过早卡在一个很刺眼的数字
- 最终还是由真实阶段信号或 `App ready` 收口

这次调整后的思路是：

- 从 `68%` 硬停改成更平滑的高位缓进
- 让用户感知成“还在继续进入系统”，而不是“卡死”

#### 2. 默认首屏不主动预取工具重资源

### 2026-04-23：CDN 熔断不能直接等于静态资源不可用

#### 现象

- 版本升级窗口里，`tool-windows-*.js` 这类带 hash 的静态资源会被 `sw.js` 接管
- DevTools 里能看到资源直接访问 CDN 是存在的
- 但页面侧请求仍然被 Service Worker 返回 `503`
- Cache Storage 里还能看到该资源已被某个静态缓存桶记录

#### 根因

问题不在“CDN 上有没有这个文件”，而在：

- `fetchFromCDNWithFallback()` 会先看 CDN 健康状态
- 某个 CDN 连续失败后会进入 cooldown，被 `getAvailableCDNs()` 排除
- 一旦 CDN 候选为空，运行时静态资源请求就只会回源站
- 升级窗口中，旧 hash 资源在源站很可能已经 `404`
- 于是链路变成：
  `CDN 被熔断跳过 -> 源站 miss -> SW 合成 503`

一句话：

**“CDN 暂时不健康”不等于“这个版本化静态资源此刻不可取”。**

#### 修复思路

更稳的做法不是放开全局熔断，而是：

- 正常请求继续遵守 CDN cooldown
- 只有在“本地源站也失败、马上要返回 503”这一刻
- 再做一次强制 CDN recovery probe

这样能同时满足两点：

- 平时不让异常 CDN 干扰正常流量
- 升级期遇到旧 hash 资源时，不会因为短期熔断把本来可用的 CDN 资源误杀

#### 经验

- 对 hash 静态资源来说，“可恢复探测”比“严格熔断”更重要
- SW 返回 `503` 前必须输出结构化诊断日志，否则很难区分是：
  - 源站 miss
  - CDN cooldown
  - recovery probe 失败
- 这类问题本质上不是资源部署问题，而是“升级期多源回退状态机”问题

这次最终固化成：

- 默认 `idle prefetch` 分组为空
- 首屏不主动拉工具运行时
- 只有用户真的点开知识库、抽屉、媒体库、同步、备份恢复等入口时，再按需加载对应分组

一句话：  
**首屏先保证壳子可用，再按交互路径精确拉工具资源。**

#### 3. 工具资源至少拆成“运行时 / 抽屉 / 对话框 / 命令”四层

这次比较实用的拆法是：

- `tool-runtime`
  - 任务恢复、字体预热、资产集成、运行时增强
- `tool-drawers`
  - 项目抽屉、工具箱、WinBox、工具服务
- `tool-dialogs`
  - 媒体库、同步、备份恢复、版本提示、性能面板
- `tool-commanding`
  - 命令面板、画布搜索

这样做的好处是：

- 打开知识库 / 项目抽屉时，不会顺带把媒体库和性能面板一起吞掉
- 打开媒体库 / 备份恢复时，也不必背整套工具窗逻辑

#### 4. 常驻工具条只依赖“抽屉组已可用”，不要依赖整个工具系统

页面监听 SW idle prefetch 完成状态时，也要监听最小必要组：

- 不要等整个工具系统 ready
- 只等能把常驻入口显示出来的那一组

这次最终收敛到：

- 常驻工具条只把 `tool-drawers` 当优化信号
- 即便预取信号丢失，也要有超时放行

一句话：  
**能显示入口，不等于要先准备完整工具生态。**

### 这轮适合固化的检查清单

- 这个大 chunk 真是首屏必需，还是只是恰好在首屏阶段被拉起？
- 启动页卡住，是业务 ready 没到，还是模拟进度自己停住了？
- 拆分 chunk 后，下面这些有没有一起改：
  - 类型定义
  - Vite 分组映射
  - idle manifest 默认组
  - 页面侧触发逻辑
  - SW 完成状态监听
  - 部署检查脚本
  - 启动包校验脚本
  - CDN / fallback 测试示例

### 一句话总结

这轮最值钱的经验不是“把 `tool-windows` 拆小了”，而是明确了两条规则：

- **启动假卡死，很多时候是进度条写法在误导用户**
- **chunk 分组一旦成为发布契约，重命名必须全链路同步**

后面再做启动优化时，先问两件事：

- 这是不是首屏真的必须？
- 如果把它改名/拆组，相关脚本和测试是不是也一起动了？

### 2026-04-24：页面 ready 不等于离线资源已完整

#### 现象

这次线上暴露的是一个更隐蔽的问题：

- 页面首屏已经能正常打开
- 切到离线模式后，壳子也还能进
- 但点开按需加载模块时，仍然会报资源未准备好

用户体感会很割裂：

- 一眼看上去像“离线已经可用”
- 真正进入知识库、媒体库、工具窗、富文本公式等懒模块时才炸

一句话：

**首屏可用，只说明启动资源够了，不代表离线工作集已经齐了。**

#### 这次最容易误判的地方

##### 1. 只盯懒加载 JS/CSS chunk，会漏掉真正缺失的附属资源

这次排查里，大多数懒加载 JS/CSS 其实已经在 `precache-manifest` 里。

真正的缺口主要是：

- 字体（`woff` / `woff2` / `ttf`）
- 未来可能出现的图片类附属资源（`png` / `webp` / `avif` 等）

也就是说：

- 模块代码本身可能已经缓存了
- 但模块运行时依赖的字体/图片没有提前准备
- 离线首次打开相关功能时，仍然会失败

##### 2. 页面侧预取分组名和构建产物分组名一旦漂移，SW 会“看起来在预取，实际上什么也没做”

这次还踩中了一个发布契约问题：

- 页面侧请求的是一套分组名
- `vite.config.ts` 产出的 `idle-prefetch-manifest.json` 却是另一套分组名

结果是：

- `SW_PREFETCH_GROUPS` 消息发出去了
- SW 也收到了
- 但 manifest 里根本没有这些组
- 预取流程形式上成功，实际上没有新增资源入缓存

一句话：

**预取分组名不是页面内部细节，而是构建产物契约。**

##### 3. 默认 idle-prefetch 命中的是“体验组”，不等于“离线完整组”

`tool-windows` 这类分组更偏体验优化：

- 目的是减少用户第一次点开工具时的等待
- 不是为了兜底全部离线静态依赖

如果把“工具分组预取完成”误认为“离线已经完整”，后面就一定会继续踩坑。

#### 这次最终收敛出来的规则

##### 1. `precache` 负责首屏壳子，`idle-prefetch` 负责首屏之后的补齐

当前更稳的职责划分是：

- `precache-manifest`
  - 负责启动必需资源
  - 目标是保证页面能稳定打开
- `idle-prefetch-manifest`
  - 负责页面 ready 后的空闲补齐
  - 目标是把首屏之外、但离线仍可能首次命中的静态资源补进缓存

不要试图只靠 `precache` 把所有离线资源都打满，也不要指望默认工具分组自动覆盖全部资源。

##### 2. 页面 ready 后要有一个明确的“补剩余静态资源”动作

这次最终采用的方案是新增一个专用分组：

- `offline-static-assets`

它的职责不是表达业务能力，而是表达缓存语义：

- 收集所有**未进入 `precache`** 的剩余静态资源
- 在页面真正 ready 后，由主线程主动请求 SW 空闲预取

这样做有两个好处：

- 不会额外放大首屏安装期的缓存成本
- 又能在用户真正切离线前，把非首屏但高概率命中的静态依赖补齐

##### 3. 构建侧必须以 `precache` 为基线做增量，不要重复扫同一批资源

这次补齐逻辑里，构建阶段先读取 `precache-manifest.json`，再生成：

- `offline-static-assets = 全量静态资源 - precache 已包含资源`

这样能避免两个问题：

- 重复预取已经在 install 阶段缓存过的资源
- 让“离线补齐”从语义上始终保持增量

##### 4. 对外暴露的分组名要尽量少，且要和构建口径共用同一套类型

这次最终没有继续扩散页面语义分组，而是收敛成：

- `tool-windows`
- `offline-static-assets`

经验是：

- 页面侧想表达“运行时 / 抽屉 / 对话框”的语义没问题
- 但一旦这些名字要穿过 SW 消息边界，就必须和构建产物完全一致
- 最好由共享类型统一导出，避免页面和构建各写一套字符串

#### 这轮适合固化的检查清单

- 页面 ready 时，离线补齐流程有没有真正发起？
- `idle-prefetch-manifest` 里新增分组，对 `precache` 的增量到底是几项？
- 离线失败时，缺的是：
  - JS chunk
  - CSS
  - 字体
  - 图片
  - 还是 SW 分组命名本身不匹配？
- 页面侧使用的预取分组名，是否和构建产物 manifest 完全一致？
- 新增静态资源类型时，是否同步评估：
  - 要不要进 `precache`
  - 还是应该进入 post-boot 补齐组

#### 一句话总结

这次最值钱的经验不是“又多预取了一组资源”，而是明确了：

- **页面 ready 只代表壳子 ready，不代表离线工作集 ready**
- **离线完整性要盯整个静态依赖面，而不只盯懒加载 JS**
- **SW 预取分组名一旦进入消息协议，就必须当作发布契约来维护**

## 补充经验：真实静态 HTML 不能走 App Shell 回退（2026-04-24）

### 现象

线上访问这类 URL 时：

- `/user-manual/index.html?board=...`
- `/user-manual/advanced-settings.html`

页面会返回首页或手册首页的 HTML，导致目标静态页面不可用。

### 这次真正踩中的根因

#### 1. 把任意 `.html` 都当成 App Shell，是错误建模

这次 SW 里曾使用类似判定：

- `request.mode === 'navigate'`
- 或 `pathname.endsWith('.html')`

这会把两类请求混在一起：

- SPA 导航请求
- 真实存在的静态 HTML 文档

像 `/user-manual/index.html` 这类页面虽然名字叫 `index.html`，本质仍是目录下的真实静态文档，不是站点根壳。

#### 2. 根壳双 key 镜像不能扩散到子目录 `index.html`

之前为了让首页导航更快命中缓存，会把：

- `/`
- `/index.html`

做双 key 映射。

但如果判断写成：

- `pathname.endsWith('/index.html')`

就会把 `/user-manual/index.html` 也误纳入根壳镜像范围，带来缓存串页风险。

### 这次应固化的规则

#### 1. App Shell 只允许覆盖根壳页

应该只把下面两类请求视为根壳：

- `/`
- `/index.html`

以及：

- **无扩展名** 的 SPA 导航路径（如 `/workspace/abc`）

下面这些都必须按真实文档处理，不能回退首页：

- `/user-manual/index.html`
- `/foo/bar/index.html`
- `/advanced-settings.html`

#### 2. 根壳 alias 只允许 `/` 和 `/index.html`

`/` 与 `/index.html` 的双 key 缓存规则，只能作用于站点根壳本身。  
不要把“任何以 `/index.html` 结尾的路径”都视为可镜像入口。

#### 3. 这类逻辑必须拆成独立 helper，并补定向单测

比起把判断散落在 `handleStaticRequest` 里，更稳的做法是：

- 单独抽出 app-shell 路由判定 helper
- 明确测试：
  - `/`
  - `/index.html`
  - `/workspace/abc`
  - `/user-manual/index.html`
  - `/advanced-settings.html`

这样以后再改 SW 缓存策略时，不容易把静态文档重新带坏。

### 这次排查里一个很值钱的动作

当发现“某个 HTML 页面返回了首页内容”时，第一步不要先认定是 SW。

应先直接请求线上 URL，确认：

- 源站本身返回的是目标页面
- 还是源站已经先 rewrite 成首页

这一步能快速把问题分成两类：

- 部署层 rewrite 错误
- SW 缓存 / fallback 错误

### 一句话总结

- **显式 `.html` 文档不是天然的 App Shell**
- **根壳缓存优化只能服务根壳，不能泛化到所有 `index.html`**
- **先区分源站直出错误还是 SW 拦截错误，再动修复**

## 补充经验：按需模块的运行时静态资源要单独成组（2026-04-24）

### 现象

页面首屏可以正常打开，但点击按需加载的功能时，仍会出现：

- JS chunk 已经下载
- 对应字体 / 图片 / wasm 等附属静态资源没有进缓存
- 离线或弱网下，按需模块仍然启动失败

表现上最容易误判成：

- “SW 预缓存没生效”
- 或 “tool-windows 预取已经完成，为什么功能还缺资源”

### 根因

#### 1. `tool-windows` 只代表代码分组，不代表运行时附属资源齐了

之前默认 idle 预取命中的是：

- `tool-windows`

这能覆盖工具相关的 JS/CSS chunk，但覆盖不到另一类关键依赖：

- `/assets/` 下的字体
- `/assets/` 下的图片
- `/assets/` 下的 wasm / 二进制附属资源

一句话：

**按需模块“代码到了”，不等于“运行时静态依赖到了”。**

#### 2. `offline-static-assets` 太宽，不能直接当默认预热组

仓库里已经有：

- `offline-static-assets`

但它表达的是“完整离线补齐”，范围很宽，里面可能包含：

- 手册截图
- 非首屏图像
- 其他 post-boot 资源

如果直接把它塞进默认 idle 预取，会带来两个问题：

- 默认预热体积膨胀
- 把“运行时必需资源”和“完整离线增强资源”混为一谈

#### 3. UI 对“预取完成”的判定过早

之前常驻工具条放行时只检查：

- `tool-windows` 是否完成

这会导致一种半完成状态：

- 代码组完成了
- 运行时附属静态资源还没完成
- UI 却已经把相关入口当作“已预热”

### 这次收敛出来的规则

#### 1. 默认 idle 预取要拆成两层语义

应该拆成：

- `tool-windows`
  - 表达按需模块代码分组
- `runtime-static-assets`
  - 表达按需模块运行时真正依赖的 `/assets/` 静态附属资源
- `offline-static-assets`
  - 继续保留为更宽的完整离线补齐组

这样职责才清晰：

- 默认 idle 只补“功能能跑起来必须有的东西”
- 更宽的离线增强资源继续延后

#### 2. `runtime-static-assets` 应只覆盖构建产物里的 `/assets/`

这次更稳的边界是：

- 只收集未进入 `precache` 的 `/assets/` 静态资源
- 不把手册截图、外部图像、说明文档等内容塞进默认组

一句话：

**默认预热只补运行时依赖，不补内容型资源。**

#### 3. 工具入口的 ready 条件必须看完整依赖组

像常驻工具条、工具窗口这类入口，如果依赖组是：

- `tool-windows`
- `runtime-static-assets`

那 UI 判定 ready 时，也必须确认这两组都完成。  
只盯代码组，会把“半预热完成”误判成“功能可用”。

### 适合固化的检查清单

- 某个按需功能失败时，先分清缺的是：
  - 代码 chunk
  - 还是 `/assets/` 下的附属静态资源
- 默认 idle 预取组，是否只覆盖“运行时必需资源”？
- `offline-static-assets` 是否仍然保持更宽的离线补齐语义？
- UI 放行条件，是否和真实依赖组完全一致？

### 一句话总结

- **按需模块预热不能只预热代码 chunk**
- **完整离线补齐组不能直接拿来当默认预热组**
- **默认预热要单独拆出运行时静态资源层，UI ready 也要按完整依赖组判断**

## 2026-04-24：跨域静态资源缓存 key 与第三方 CDN 路径要分开处理

这轮线上问题还有两个更隐蔽的坑：

### 现象

- `Cache Storage` 里已经能看到某些 `/assets/...` 文件
- 但切到离线模式后，请求 `https://cdn.jsdelivr.net/npm/aitu-app@<version>/assets/...`
  仍然被 SW 返回 `503`
- 某些第三方资源还会出现这种错误回退：
  - 期望：`https://cdn.jsdelivr.net/npm/winbox@0.2.82/dist/...`
  - 实际被拼成：`https://cdn.jsdelivr.net/npm/aitu-app@<appVersion>/npm/winbox@0.2.82/dist/...`

### 根因

#### 1. 预缓存写入 key 和运行时读取 key 不一致

预热/预缓存时，为了统一源站与 CDN 地址，会把静态资源写入成“同源归一化 key”，例如：

- 请求地址：`https://cdn.jsdelivr.net/npm/aitu-app@0.6.67/assets/a.js`
- 实际缓存 key：`https://pr.opentu.ai/assets/a.js`

如果运行时读取仍只按原始请求 URL 查：

- `cache.match(request)`

那离线时就会 miss，哪怕缓存里已经有同内容。

#### 2. 第三方 npm 包路径不能复用 `aitu-app` 模板

像这类资源：

- `/npm/winbox@0.2.82/dist/winbox.bundle.min.js`

它本身已经是“第三方包路径”。

如果还按自家应用资源那套规则去拼：

- `https://cdn.jsdelivr.net/npm/aitu-app@{version}/{path}`

就会把第三方路径错误地套进 `aitu-app@版本` 目录下，必然 404 或离线兜底失败。

### 这次固化出来的规则

#### 1. 静态资源缓存读取必须双查

运行时静态资源命中缓存时，至少要查两把 key：

- 原始请求 URL
- 归一化后的同源 cache key

并且回查旧版本 cache 时，也要用同一套 key 规则。

#### 2. 运行时写缓存也要统一落到归一化 key

不然就会出现：

- 预缓存写一套 key
- 运行时补缓存再写另一套 key

最后同一份资源在不同 key 下分裂，离线行为不可预测。

#### 3. 第三方 npm 资源要单独识别

对于 `/npm/<pkg>@<version>/...` 这类第三方资源：

- 不要套 `aitu-app@APP_VERSION`
- 应该直接构造真实第三方 CDN 地址
- 本地源站 fallback 也只能作为兜底，不能改写包名和版本语义

### 适合固化的检查清单

- 线上静态资源如果“明明缓存里有，离线却 503”，先查：
  - 写缓存的 key 是什么
  - 读缓存时查的是不是同一把 key
- 运行时 `cache.put` 和安装阶段 `cache.put` 是否共用同一套归一化规则？
- 第三方 npm 包 URL 是否被错误套进业务应用包模板？
- `cdn.jsdelivr.net/npm/<pkg>@<ver>/...` 这类请求，回退逻辑是否保留了原包名和版本？

## 2026-04-24：完整离线补齐不能只依赖页面侧 postMessage

这轮还有一个更隐蔽的问题：

### 现象

- 首阶段静态资源会加载到一个固定数量后停住
- 浏览器里能看到默认 idle 预取已经完成
- 但更宽的 `offline-static-assets` 没继续补齐

### 根因

之前的链路是：

- `SW activate` 后自动拉默认组
- 页面启动完成后，再由前端 `postMessage` 请求 `offline-static-assets`

这个设计有两个脆弱点：

#### 1. 第二阶段依赖页面消息成功送达

如果页面当时还没拿到 `controller`，或者消息发送时机不稳，第二阶段就根本不会开始。

#### 2. 页面侧有本地去重，失败后可能不会再发

前端为了避免重复请求，会把已请求的 group 记在内存里。  
如果第一次发送实际上没送到 SW，但本地已经记成“发过了”，后面就会出现：

- 页面以为请求过了
- SW 实际没收到
- 资源补齐永久停在半路

### 这次固化出来的规则

#### 1. 第二阶段离线补齐要由 SW 自己接力

像 `offline-static-assets` 这种“完整离线增强组”，不应该只靠页面再发一次消息触发。  
更稳的做法是：

- 默认组完成后
- SW 自己按空闲窗口继续调度第二阶段

这样就算页面消息丢了，后台也会继续补齐。

#### 2. 页面消息只能当增强触发，不能当唯一触发源

前端 `postMessage` 仍然可以保留，用于更早拉起第二阶段。  
但真正的兜底必须在 SW 内部。

#### 3. 去重状态必须能回收

如果页面侧发送失败：

- 不能一直把 group 留在“已请求”集合里
- 否则后续重试机会会被自己吃掉

### 适合固化的检查清单

- 默认 idle 预取完成后，SW 是否还会自动接力更宽的离线补齐组？
- 第二阶段是否只依赖页面 `postMessage`？
- 页面侧去重集合在发送失败时，是否会正确释放？
- 调试日志里是否能直接看出：
  - 默认组完成
  - follow-up 组已排队
  - follow-up 组已开始

## 版本目录回滚与 SW Scope 经验

更新日期：2026-04-25

### 现象

发布 `0.6.76` 后，后台仍然疯狂下载 `0.6.74` 的资源，升级提示也显示 `0.6.74`，但进入应用后版本号又是 `0.6.76`。

这类问题本质上是：

- 页面入口、`version.json`、SW、预热清单不在同一个版本目录语义下解析
- 旧 SW 可能消费了新目录或根目录的清单
- CDN fallback 又用旧 `APP_VERSION` 拼出了旧包资源地址

最终表现为：

- UI 认为自己是新版本
- SW 后台按旧版本预热
- CDN 日志里出现旧 npm 包版本资源

### 根因

#### 1. 根路径注册 SW 会破坏 release 目录隔离

如果在 `https://opentu.ai/releases/0.6.68/` 页面里执行：

```ts
navigator.serviceWorker.register('/sw.js')
```

注册的是根路径 `/sw.js`，而不是当前 release 目录里的 `./sw.js`。

这会导致：

- 回滚目录无法拥有自己的 SW scope
- 根 SW 可能控制多个 release 页面
- `version.json`、manifest、App Shell cache key 容易跨版本串读

正确做法是：

```ts
navigator.serviceWorker.register('./sw.js')
```

让每个 release 目录注册自己的 SW，并把 scope 固定在当前目录。

#### 2. SW 内部不能默认用 origin 根目录读取清单

这些写法在根部署时看起来没问题：

```ts
fetch('/version.json')
new URL('/idle-prefetch-manifest.json', location.origin)
cache.match('/')
cache.match('/index.html')
```

但在 `/releases/{version}/` 下会越过当前版本目录。

SW 内部应基于自身脚本位置创建 scope base：

```ts
const SW_SCOPE_BASE_URL = new URL('./', self.location.href);
```

所有入口清单、App Shell alias、本地回源地址都应基于这个 base 解析。

#### 3. 不能用“低于已发布版本”阻断安装

看似合理的保护逻辑是：

- 如果 SW 版本低于线上最新版本，就中止旧 SW 安装或预热

但这会破坏主动回滚：

- `0.6.76` 发布有问题时，需要能访问 `/releases/0.6.68/`
- 旧版本目录的 `version.json` 本来就应该是 `0.6.68`
- 不能因为它小于当前最新版本就拒绝安装

正确规则是：

- **允许回滚到任意明确版本目录**
- **只拒绝当前 SW 消费与自身 `APP_VERSION` 不一致的 manifest**
- 不按 semver 大小判断“是否允许安装”

### 修复策略

- SW 注册统一使用相对路径 `./sw.js`。
- 升级弹窗读取 `./version.json`，避免读取根目录版本信息。
- SW 内部新增 scope base，基于当前 SW 所在目录解析：
  - `version.json`
  - `precache-manifest.json`
  - `idle-prefetch-manifest.json`
  - App Shell alias `/` 与 `index.html`
  - CDN fallback 的本地源站兜底地址
- manifest 加载后校验 `manifest.version === APP_VERSION`，不一致则跳过该清单。
- 不引入“低版本 SW 直接中止”的逻辑，保留版本目录回滚能力。

### 后续守则

- 支持版本目录发布时，禁止在前端启动链路里使用根绝对 SW 注册路径。
- SW 里凡是代表“当前应用入口”的 URL，都要问一句：这是 origin 根路径，还是当前 SW scope？
- CDN URL、同源 URL、release 目录 URL 必须先归一为 scope-relative 资源路径，再生成 cache key。
- 升级提示只应忽略“版本相等”，不要因为版本号更小就压制提示或安装；更小版本可能是有意回滚。
- 预热清单是版本绑定资产，旧 SW 只能消费旧 manifest，新 SW 只能消费新 manifest。
