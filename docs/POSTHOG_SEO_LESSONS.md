# PostHog 埋点观测与 SEO 优化经验总结

这篇文档沉淀 2026-04-21 这轮工作的经验，重点包括两部分：

- 如何判断“埋点是否真的生效”
- 如何在不新增实体页的前提下，提升 SEO 搜索触达率

## 一、PostHog 埋点观测经验

### 1. 先看业务动作埋点，不要只盯标准事件

这次观察里，业务动作埋点已经明显生效，但 `$ai_generation` 仍然为 0。

已确认有数据的关键事件包括：

- `ai_input_click_send`
- `task_click_insert`
- `task_click_download`
- `task_click_retry`
- `generation_result_insert_canvas`
- `generation_result_download`
- `generation_retry_after_failure`
- `task_recovered_after_reload`

结论：

- “今天基于 MCP 补的业务埋点”已经打通
- “PostHog LLM starter 标准事件”还没有真正接入当前主生成链路

经验：

- 埋点验证要分层看
- 业务链路埋点正常，不代表标准分析事件也正常
- 标准事件为 0，不代表整条埋点链路都失败

### 2. 观察当天数据时，优先用高意图动作判断是否生效

比起只看曝光类事件，更建议优先看这些“强意图动作”：

- 发送生成
- 插入画布
- 下载结果
- 重试失败任务
- 刷新后任务恢复

原因：

- 这些动作更接近真实使用
- 一旦有值，基本可以证明核心链路已通
- 对判断“埋点是否落地成功”比页面浏览类事件更直接

### 3. 埋点复盘要输出“哪部分通了、哪部分没通”

只说“有数据”不够，必须明确拆分：

- 哪些业务事件已经生效
- 哪些标准事件仍缺失
- 问题属于“埋点未发出”还是“链路未接入”

推荐复盘口径：

- 业务动作埋点：已通
- 通用性能/页面监控：另看
- `$ai_generation` 标准链路：待补

这样后续继续修埋点时，范围更清晰，不会重复排查已通过的部分。

## 二、SEO 优化经验

### 1. 不新增页面实体，先吃透现有页面

本轮优化遵循：

- 如无必要，不新增页面
- 优先改已有首页、手册页、功能页
- 通过标题、描述、H1、正文锚文本、内链来扩展搜索覆盖

经验：

- 现有页面如果没有吃满意图，继续新建页面往往只会分散权重
- 先把已有页面的搜索意图做清楚，收益更快

### 2. 首页不要硬堆词，要按页面角色分词

推荐分工：

- 首页：品牌词、平台词、总入口词
- 功能页：模型词、能力词、操作词
- 手册页：教程词、问题词、场景词

本次采用的策略：

- 首页承接 `AI应用平台`、`画布工作区`、`流程图`、`思维导图`
- 图片页承接 `文生图`、`参考图`、`批量生图`、`nano-banana`
- 视频页承接 `文生视频`、`图生视频`、`Veo3`、`Sora`
- 拆图页承接 `九宫格拆图`、`朋友圈宫格图`
- 流程图页承接 `Mermaid 转流程图`
- 思维导图页承接 `Markdown 转思维导图`

经验：

- 同一批关键词不要全堆到首页
- 应该让“页面内容”和“搜索意图”一一对应

### 3. 模型词和应用词会提升触达率，但必须落在正文里

用户可能按以下方式搜索：

- 按模型搜：`nano-banana`、`Veo3`、`Sora-2`
- 按应用搜：`批量生图`、`图生视频`、`九宫格拆图`
- 按操作搜：`Markdown 转思维导图`、`Mermaid 转流程图`

经验：

- 只放在 `meta keywords` 没用
- 只放在首页一处也不够
- 最有效的是把词落在目标页正文、标题、H1、真实锚文本中

### 4. 内链优化比继续堆 meta 更值钱

本轮后续优化的核心不是继续改 meta，而是补正文内链。

高价值内链策略：

- `素材库` → `advanced-media-library.html`
- `API 配置 / API Key` → `advanced-settings.html`
- `批量生成 / 批量生图` → `advanced-toolbox.html`
- `九宫格拆图` → `ai-generation-image-split.html`
- `灵感创意板` → `ai-generation-inspiration-board.html`
- `AI 生成图片` → `ai-generation-image-generation.html`
- `AI 生成视频` → `ai-generation-video-generation.html`
- `Markdown 转思维导图` → `drawing-mindmap.html`
- `Mermaid 转流程图` → `drawing-flowchart.html`
- `导出与导入` → `advanced-export-import.html`

经验：

- 正文里的真实锚文本，比孤立的导航链接更有语义价值
- 内链要让“能力页”和“场景页”互相导流
- 用户读到哪里、下一步要做什么，就把链路接到哪里

### 5. 手册首页既是导航页，也是 SEO 入口页

手册首页不能只是目录。

更好的做法是：

- 用一段总述解释平台定位
- 把核心能力写成结构化列表
- 在列表里挂上高价值关键词和功能页内链

本次首页整理后的经验：

- 先修不规范 HTML 结构
- 再把“模型词 + 功能词 + 页面跳转”一起补进去

这样首页就不仅是目录，也会成为更强的入口页。

## 三、推荐工作流

以后再做同类工作，建议按这个顺序执行：

1. 先用 PostHog 看当天核心业务动作是否有值
2. 明确“业务埋点已通”还是“标准事件已通”
3. 首页补品牌词和平台级能力词
4. 功能页补模型词、场景词、操作词
5. 手册页补正文内链，不继续机械堆 meta
6. 最后用 `rg` 快速检查 `title`、`meta description`、`h1`、`href`

## 四、这轮工作的关键结论

### 埋点侧

- 今天补的业务埋点已生效
- `$ai_generation` 仍需后续独立修主链路接入

### SEO 侧

- “支持的模型 + 应用场景关键词”确实会提升触达率
- 但必须放在正确页面的正文和内链里
- 现阶段继续做内链优化，比再堆一轮 meta 更有收益

## 五、一句话原则

### 埋点

先确认业务动作有没有被记录，再确认标准事件有没有接上。

### SEO

不要让首页替所有页面吃词，要让每个页面只吃自己最该吃的搜索意图。

## 六、SEO 首页落地页设计经验（OpenTu home.html）

这次新增 `home.html` 的目标不是再堆一批重复 SEO 页面，而是把品牌首页做成一个更可信、更可转化的统一入口。

### 1. 面向用户展示卖点，不展示内部讨论过程

页面文案必须站在客户视角表达价值。

不应出现在页面上的内容：

- “README 展示案例”
- “PostHog”
- “为什么强调某模型”
- “这是为了 SEO 增加的页面”

这些属于内部策略和实现细节，只适合写在文档、PR 或复盘里。

推荐页面表达：

- 产品是什么
- 能解决什么创作断点
- 用户下一步可以怎么开始
- 为什么值得信任，例如开源、可自部署、作品可沉淀

经验：

- 内部策略可以指导页面结构，但不能成为用户可见文案
- 用户只关心收益，不关心我们为什么这样设计

### 2. 首页首屏只保留一个核心主张

首屏最容易犯的问题是信息过多：标题、副标题、标签、卖点卡、按钮、示例图全部抢注意力。

本次最终收敛为：

- H1：`开源 AI 创作台`
- 一段短副标题解释画布价值
- 三个轻量卖点卡
- 两个 CTA：进入画布、查看开源项目
- 一个真实产品动图作为视觉证据

经验：

- H1 不要承担全部解释任务，只负责把定位讲清楚
- 副标题只补充使用场景，不继续堆关键词
- 卖点卡要短，更多像“信任锚点”，不是详细说明书
- CTA 与卖点的顺序会影响视觉重心；卖点卡在前、按钮在后更适合弱化强推销感

### 3. 标题要短，SEO 关键词放到页面其他位置承接

为了视觉专业度，H1 不宜过长。

这次没有把 H1 写成：

- `OpenTu AI 创作工作台 | GPT-Image-2 与 DeepSeek-v4 画布`
- `开源 AI 图片生成与知识画布工作台`

而是收敛为：

- `开源 AI 创作台`

SEO 关键词仍然可以放在：

- `title`
- `meta description`
- section 标题
- FAQ
- JSON-LD
- 正文段落
- 内链锚文本

经验：

- H1 负责“人读起来顺”
- SEO 关键词由整页结构共同承接
- 不要为了搜索词牺牲首屏可信度

### 4. 首屏视觉要控制纵向留白和信息密度

本次设计中间经历了几个问题：

- 顶部导航和首屏距离过大，导致首屏内容下沉
- 标签、标题、副文案、按钮、卖点卡堆在一起，左侧头重脚轻
- 首屏与第二段之间留白过大，页面像断层
- 标题字距过紧，中文大字号显得压迫

最终调整方向：

- 减少导航底部 padding
- 减少 hero 底部 padding
- 单独控制 `.hero + .section` 的间距，避免影响其他 section
- 大标题适当放大字距，提升高级感
- 桌面端标题尽量单行，小屏允许自然换行

经验：

- 首页首屏的“专业感”往往来自间距，而不是更多装饰
- 不要只看单个模块，要看首屏和下一段之间的节奏
- 中文大标题需要比英文更谨慎地控制字距和换行

### 5. 示例图要承担视觉证据，不要只做装饰

右侧示例图使用 README 中已有的产品动图，避免旧图和页面卖点不一致。

经验：

- 示例图应直接证明页面卖点，例如画布、作品整理、生成过程
- 不要首屏放与当前产品形态不一致的旧图
- 图片浮层文案要短，只做补充，不抢 H1 和 CTA

### 6. SEO 页也要有可分析的行为埋点

`home.html` 虽然是静态落地页，但仍需要埋点判断 SEO 效果。

本次覆盖的行为包括：

- 页面访问
- 导航点击
- 首屏 CTA 点击
- 模型卡点击
- 作品卡点击
- 滚动深度

经验：

- SEO 效果不能只看访问量，还要看访问后的意图动作
- CTA 点击和滚动深度比单纯 PV 更能判断页面质量
- 静态页埋点要轻量，失败时不能影响页面加载

### 7. 重复 SEO 页面能删就删，用 301 合并权重

本次删除了仅为 SEO 增加、内容重复度高的页面，并将旧地址 301 到 `home.html`。

经验：

- 重复页面会分散维护成本和搜索权重
- 如果页面无法承载独立搜索意图，应合并到更强入口页
- 删除页面时必须同步更新 sitemap、redirects 和部署平台重定向配置

### 8. 首页提前注册 Service Worker 可以改善后续体验

`home.html` 会在空闲时注册 `sw.js`，让用户进入画布工作台前提前准备资源。

经验：

- SEO 落地页不只是“被搜索到”，还要帮助后续工作台加载更顺滑
- 预热资源应放到空闲时机，避免影响首屏性能
- SW 注册失败必须静默处理，不能影响用户浏览

### 9. 首页设计检查清单

以后再做类似首页，可以按这个清单检查：

1. H1 是否一句话说清产品定位
2. 首屏是否只有一个核心主张
3. 内部策略词是否全部移出用户可见文案
4. CTA 是否明确且不抢走全部视觉重心
5. 卖点卡是否足够短
6. 示例图是否能证明产品能力
7. 首屏和下一段之间是否存在断层
8. 埋点是否覆盖访问、点击、滚动和核心转化
9. sitemap 与重定向是否同步更新
10. SEO 关键词是否分布在整页结构里，而不是硬塞进 H1
